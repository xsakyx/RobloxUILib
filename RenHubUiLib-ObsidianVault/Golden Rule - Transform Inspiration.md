# Golden Rule — Transform Inspiration

> Study why a reference feels excellent, then rebuild those principles into RenLib's own coherent language. Never stop at copied names, copied layouts, or surface-level resemblance.

For every inspiration-led feature, ask:

1. What underlying problem does the reference solve: hierarchy, navigation, legibility, feedback, density, delight, or recovery?
2. Which parts belong to the reference's brand and must not be copied?
3. How can the idea integrate with RenLib's themes, responsive rules, cleanup, configuration, and existing API?
4. What improvement makes the RenLib version meaningfully more useful instead of merely different?
5. Does the result still feel like one product when the reference is removed from view?
6. Did we document the principle we learned rather than treating the screenshot as a permanent crutch?

## Definition of transformed

- Names, visual tokens, and interaction language belong to RenLib.
- The implementation is written for RenLib's architecture instead of transplanted from another library.
- The feature works across solid and frosted materials, themes, input methods, and viewport sizes.
- The feature has a clear reason to exist beyond matching a screenshot.
- The result can evolve independently without remaining tied to the reference.

This rule works together with [[Golden Rule - Feature Completion]]. Originality without completion is frustrating; completion without originality produces a library with no identity.
